## 1. Стратегия валидации и воспроизводимость

### Стратегия валидации с учетом природы данных
Классические методы перемешивания типа K-Fold CV (cross validation) не подойдут. Их принцип - данные перемешиваются, 
делятся на k фолдов. Модель обучается на K-1 частях, тестируется на последней. Здесь не подойдет, т.к. bandit-алгоритм 
в сфере моды должен быть привязан к сезону. Если перемешать все данные условно за зиму, лето, весну, то получится каша. 
Это не поможет.

Используем Time-based Split + OPE (offline policy evaluation)  
- Time-based Split - Мы не перемешиваем данные, а делим их по хронологии. У нас есть данные недельного эксперимента.
Поэтому данные с 1-4 дней отдаем на Train, 5 - validation, 6-7 - Test
- OPE - то есть мы оцениваем новую рекомендат. политику используя только старые логи, не запуская в прод. Используем IPS
и DR (Double robust). DR сочетает в себе прямой подход и IPS. V_DR = Предсказание_модели + IPS_поправка_на_ошибку_модели

## 2. Анализ утечек данных

- Временная утечка - использование будущих данных при обучении.  
Решение: Строгий time-based split, валидация по времени
- Propensity score - мета-информация о политике.  
Решение: Исключить из списка фич. Использовать только для OPE
- Позиция товара сильно влияет на CTR. Модель может выучить позицию вместо качества товара.  
Решение: добавлять позицию как контекст, но при инференсе использовать дефолтную или рандомизированную позицию для оценки

## 3. Масштабирование

### Оценка нагрузки
Исходя из датасета ZOZO (~27 млн событий за 7 дней) и предположений о продакшене:

| Параметр         | Оценка                                      | Обоснование                                                  |
|------------------|---------------------------------------------|--------------------------------------------------------------|
| RPS (средний)    | 50–100 req/s                                | 27M событий / 7 дней / 86400 сек ≈ 45 событий/сек, с запасом ×2 |
| RPS (пиковый)    | 300–500 req/s                               | Вечерние часы, распродажи, маркетинговые активности          |
| Трафик (средний) | ~150 KB/запрос × 100 RPS ≈ 15 MB/s          | JSON с контекстом + топ-10 рекомендаций                      |
| Latency (P95)    | ≤ 200 мс                                    | Пользователь не должен ждать рекомендации при скролле каталога |
| Latency (SLO)    | 99% запросов < 500 мс                       | Бизнес-требование для сохранения конверсии                   |

### Стратегия масштабирования
| Компонент            | Метод масштабирования                                      | Обоснование                                                                 |
|----------------------|------------------------------------------------------------|-----------------------------------------------------------------------------|
| API Gateway          | Горизонтальное (K8s HPA), кэширование в Redis              | Stateless, легко реплицируется; кэш снижает нагрузку на 30–50%              |
| Recommendation Service | Async inference + local LRU-cache (1000 пользователей)   | Bandit-inference быстрый (матричные операции), но кэш спасает при повторных запросах |
| Bandit Engine        | Шардирование по `item_id` + async updates                  | Обновления статистики не блокируют инференс; шарды позволяют распределить память |
| Feature Store        | Read replicas + eventual consistency                       | Инференс читает часто, пишет редко; допустима небольшая задержка обновления фич |
| Queue (Kafka)        | Увеличение партиций + консьюмер-групп                      | Обработка пиков без потери событий; replay для переобучения                 |
